# FileGramBench Pilot Evaluation Framework

## Overview

This framework evaluates 7 baseline memory systems + FileGramOS Simple on profile reconstruction from file-system behavioral trajectories.

**Core hypothesis**: Dialogue-memory systems (Mem0, Zep, MemOS, MemU, EverMemOS) perform poorly on file-system trajectories because they don't understand file operation semantics. FileGramOS Simple, with dimension-aware feature extraction, should significantly outperform them.

## Architecture

```
bench/
├── baselines/
│   ├── base.py                 # BaseAdapter interface (ingest/infer_profile/reset)
│   ├── registry.py             # Adapter registration and lookup
│   ├── full_context.py         # Lower-bound: dump everything into context
│   ├── naive_rag.py            # Chunk + keyword-retrieve + infer
│   ├── mem0_adapter.py         # Mem0: events -> pseudo-conversation -> memory
│   ├── zep_adapter.py          # Zep: events -> entity-relationship graph
│   ├── memos_adapter.py        # MemOS: events -> hierarchical memory cells
│   ├── memu_adapter.py         # MemU: events -> typed memory units
│   ├── evermemos_adapter.py    # EverMemOS: events -> engram segments
│   └── filegramos_simple.py    # Our method: deterministic features -> LLM synthesis
├── filegramos/
│   ├── feature_extraction.py   # Step 1: per-attribute deterministic extractors
│   ├── aggregation.py          # Step 2: cross-trajectory aggregation
│   └── synthesis_prompt.md     # Step 3: LLM synthesis prompt template
├── evaluation/
│   ├── judge_scoring.py        # LLM-as-Judge implementation
│   ├── judge_prompt.md         # Judge prompt template and rubric
│   ├── mcq_generator.py        # Auto-generate MCQs from trajectories
│   └── mcq_template.md         # MCQ evaluation prompt format
├── run_eval.py                 # Main evaluation runner
└── README.md                   # This file
```

## Evaluation Pipeline

### Prerequisites

60 trajectories in `signal/{profile_id}_{task_id}/events.json`
(6 profiles x 10 tasks, generated by FileGram Phase A)

### Step 1: Generate Inference Prompts (dry run)

```bash
python -m bench.run_eval --dry-run
```

This generates prompt files in `bench/scores/` without calling any LLM.

### Step 2: Full Evaluation

```bash
python -m bench.run_eval
```

### Step 3: Inspect Results

Results are organized as:
- `bench/scores/{method}_scores.json` — per-attribute LLM-as-Judge scores
- `bench/mcq/{profile_id}_mcqs.json` — generated MCQ questions
- `bench/eval_results.json` — all method outputs

## Adapter Interface

All adapters implement `BaseAdapter`:

```python
class BaseAdapter(ABC):
    def ingest(self, trajectories: list[dict]) -> None: ...
    def infer_profile(self, profile_attributes: list[str]) -> dict: ...
    def reset(self) -> None: ...
```

Adapters convert file-system events into their system's native format, then produce an inference prompt. The actual LLM call is done by the runner.

## Two Evaluation Methods

### 1. LLM-as-Judge (Per-Attribute)

- Judge sees ground truth profile + inferred profile
- Scores each attribute 1-5 with justification
- Results aggregated: per-attribute mean, per-method mean, variance

### 2. Trajectory MCQ

- Questions auto-generated from actual event data
- Ground truth from trajectories, NOT from profile YAML
- Categories map to profile attributes
- Tests factual recall of behavioral details

## FileGramOS Simple vs Baselines

| Aspect | Baselines | FileGramOS Simple |
|--------|-----------|-------------------|
| Input to LLM | Raw event narratives or system-processed memories | Pre-computed attribute statistics |
| Feature alignment | LLM must figure out what to look at | Features explicitly aligned to attributes |
| Cross-trajectory | Single trajectory or chunked | Aggregated across all 10 trajectories |
| Domain awareness | Generic dialogue/interaction patterns | File-system-specific features (depth, naming, edit magnitude) |
